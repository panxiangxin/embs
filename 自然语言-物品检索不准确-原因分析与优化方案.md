# 自然语言-物品检索不准确：原因分析与优化方案

## 0. 现状梳理（以本仓库为例）

当前项目主要分两块：

- `app.py`：提供本地 Embedding API（默认 `BAAI/bge-small-zh-v1.5`），用于把文本转成向量。
- `demo/app.js`：在浏览器侧做“物品/资产检索”的主要逻辑，支持两种检索模式：
  - **Label 级（label mode）**：对每个 label 建向量；检索时先做候选召回（topM label），再对候选资产做打分。
  - **Entity 级（entity mode）**：把一个资产的多个 labels 聚合成一个 entity 向量；检索时直接 query 向量 vs entity 向量相似度排序。

`demo/app.js` 里已经包含了一些关键手段：`normalizeLabel`（文本归一化）、`SYNONYMS`（同义词映射）、`IDF` 加权、颜色降权（`typeCoef`）、覆盖度式打分（多短语累计）以及 head gate（`pickHeadPhrase` + `TAU_HEAD`）。

如果你现在“自然语言→物品检索不准确”，大概率不是单一原因，而是**数据（labels）× query（自然语言）× 召回/打分策略 × 模型**共同造成的。

---

## 1. 为什么会不准确：高频根因清单

### 1.1 数据侧：labels 本身不适合被检索

常见问题：

- **标签不全**：物体没有“类型词”（如“房子/建筑/门/箱子/卡车”），只有属性词（如“红色/破损/金属”）。
- **标签粒度不一致**：有的用“货车”，有的用“卡车”，有的用“车辆”，导致语义空间里距离不稳定。
- **同义词/别名未归一**：如“屋子/房子/建筑”，“铁门/大门/门”，“皮卡/货车/卡车”等。
- **过多通用词**：颜色、材质、大小等高频标签覆盖面太大，容易“劫持”检索结果。

直接后果：即使 embedding 很好，检索也会变成“谁更像红色/金属”，而不是“谁是卡车/门/塔”。

### 1.2 Query 侧：自然语言里信息噪声多、目标描述没被正确抽取

自然语言指令通常包含：

- 动作/意图（“拿起/打开/去/朝着/标记/攻击”）
- 指代（“那个/这边/右边/前面”）
- 修饰（“看起来像/有点/可能/附近”）

如果把整句直接拿去 embedding 检索，会把很多**非目标信息**混进 query 向量里，导致相似度偏移。

> 经验上：检索阶段输入最好是“目标物体描述短语”，而不是完整指令句。

### 1.3 短语抽取策略导致“只剩下属性词”

以 `demo/app.js` 的 `extractPhrases(text, vocab)` 为例：它会优先从 query 中“命中 vocab 的 label 子串”并返回这些 phrases。

风险点：

- 当 query 里包含**未在 vocab 出现的关键名词**时（例如“房子”不在 labels 里），抽取可能只保留下“红色/破损”等属性词。
- 结果就是 head gate 也会退化（head 可能只剩颜色），最终变成“颜色匹配就能过”，误召回增多。

典型现象：`"红色的房子"` 可能匹配到“红色卡车”，因为系统只识别到“红色”。

### 1.4 召回/打分策略不足以表达“多约束”

如果你的线上实现仍是下面这种（非常常见）：

- `score(entity) = max_{label in entity.labels} sim(query, label)`

那么只要某个通用 label（比如“红色”）相似度很高，就可能赢过真正类型词匹配的候选。

即使你像 demo 那样做了“多短语覆盖度打分”，也仍然会因为：

- 阈值（`TAU_COVER` / `TAU_HEAD`）没调好
- `topM` 太小导致召回不足
- 权重（IDF/类型系数）对你的数据分布不合适

而出现“看起来随机/不稳定”的检索结果。

### 1.5 Embedding 模型与领域不匹配 / 用法不匹配

常见表现：

- 通用模型对“游戏内物品/资产标签”的区分度不足（尤其是短 label、行业词、项目内自造词）。
- BGE/对比学习类检索模型通常会有“query/doc prompt”的最佳实践；如果 query 和 label 使用方式不一致，效果会掉。

## 2. 优化方案（按投入产出分层）

下面给的是“能落地”的改法，从 P0 到 P2 逐步增强；不需要一次做完。

### P0（当天可做）：低成本立刻提升稳定性

1) **把“目标描述”从整句指令中抽出来**
- 规则法起步：去掉常见动作词、礼貌词、虚词（“请/帮我/一下/那个/这边/去/拿/打开…”），保留名词短语 + 属性短语。
- 或接入 LLM/意图识别：输出结构化结果 `{intent, target, attributes, relations}`，检索只用 `target + attributes`。

2) **标签与 query 做同一套强归一化**
- 统一：大小写、全角半角、标点/多空格、数字单位、繁简（如有）。
- 维护“同义词/别名词典”：至少覆盖你们游戏里 top 200 高频物体类型词与属性词。
- 把复合标签拆成原子标签（例：“红色卡车”→`红色`+`卡车`）。

3) **短语抽取别丢掉“未知名词”**
- 建议策略：`phrases = 命中labels + 重要剩余片段 + 全句fallback`
  - 命中 labels：用于精确约束（可加权）。
  - 未命中但疑似“类型词”的 token：不要因为没在 vocab 就丢弃。
  - **始终追加一个全句/长短语的 embedding** 作为兜底，避免只剩属性词。

4) **强制“类型词优先”的 head gate**
- head 只从“类型词候选”里选（颜色/材质/大小不算 head）。
- 当 query 没抽到类型词时，不要用颜色当 head gate；改为：
  - 降级为 entity mode；
  - 或直接提示“目标不明确”走交互澄清。

5) **加可解释日志，先把问题看清**
- 每次检索至少记录：抽取的 phrases、每个 phrase 命中的 best label、覆盖数、最终分数、`Ctext`（top1/top2 margin）。
- 你会很快定位到底是“抽取错了”还是“labels 不够”还是“模型拉不开”。

### P1（1~3 天）：召回更全、排序更准

1) **两段式检索：label 召回 + entity 重排（强烈推荐）**
- 召回：label 向量 topM（或 BM25 topM）→ 候选实体集合 `Cand`。
- 重排：对 `Cand` 做覆盖度打分（多短语累计）+ head gate + 规则加权（如颜色/材质降权）。

2) **引入混合检索（Hybrid）**
- 向量检索擅长“语义同义”；BM25/字面匹配擅长“专有名词/缩写/编号”。
- 可用简单线性融合：`final = α * score_vec + (1-α) * score_bm25`，α 从 0.6~0.8 起试。

3) **阈值与权重用数据驱动调参**
- 用你们自己的 query 样本（哪怕 50 条）做小评测，调 `topM / TAU_COVER / TAU_HEAD / 颜色系数`。
- 实操建议：先固定召回足够大（topM 50→200），再调阈值降低误召回。

4) **补齐 labels：让每个物体至少有一个“类型词”**
- 建议为每个资产强制有：`{类别词, 子类别词, 属性词(可选), 别名(可选)}`。
- 例：门：`门/大门/铁门/钢制/破损`；车：`车辆/卡车/货车/红色/重型`。

### P2（1~2 周）：领域化与多模态兜底（上限更高）

1) **Reranker（交叉编码器）做最终排序**
- 先向量召回 topK（如 100），再用 reranker 对 `(query, label/description)` 打分排序，精度明显更稳。

2) **Embedding 领域微调**
- 用你们的同义词、点击日志、人工标注对（query→正确实体）做对比学习微调，解决“项目内自造词/资产命名习惯”。

3) **低置信度时多模态兜底**
- 当 `Ctext` 很低（top1/top2 分数差很小）：
  - 走澄清提问；或
  - 走 CLIP（text→image）对实体截图/渲染图做检索，融合排序。

---

## 3. 建议的“最小评测闭环”（否则很难稳定提升）

1) **建立小型标注集**（50~200 条就够起步）
- 每条包含：`query`、期望 `entity_id`、可选的“允许等价集合”（同类多个都算对）。

2) **定义指标**
- `Recall@K`（K=1/3/5/10）
- `MRR@K`（排名越靠前越好）
- 统计“触发澄清/触发多模态”的比例（是否合理）

3) **把调参写成脚本**
- 固定数据集 + 可重复跑，才能知道某个改动到底有没有提升。

## 4. 针对本仓库可直接动手的点（如果你要落地在 demo）

对应文件：`demo/app.js`

- `extractPhrases`：不要只返回命中 labels 的片段；建议追加“剩余 token”与“全句 fallback”。
- `pickHeadPhrase`：head 必须来自“类型词候选”；当没有类型词时禁用 head gate 或改用澄清策略。
- `SYNONYMS` / `COLOR_LABELS`：补齐你们游戏内真实同义词与颜色/材质/形状集合；并把规则变成可配置数据文件（JSON）。
- 给 `runSearch` 增加更详细的日志输出（匹配解释），用于调参定位。
